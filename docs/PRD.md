## 1. 文档概述

### 1.1 目的

本文档详细描述 OpenStay（开放宿联）产品的功能需求、技术规格和用户体验设计，为开发团队提供清晰的实现指南。

### 1.2 目标读者

- 产品经理与设计师
- 前端/后端开发工程师
- 测试工程师
- 运营与商务团队

### 1.3 术语表

| 术语 | 说明 |
|------|------|
| **Property** | 旅店/民宿/宾馆，即住宿服务提供方 |
| **Hub** | 聚合平台，负责审核、展示、分发多个 Property 的房源 |
| **Studio** | 单店 Blocklet，Property 的独立数字店面 |
| **Traveler** | 旅客/住客，平台的终端用户 |
| **DID** | Decentralized Identity，去中心化身份 |
| **Blocklet** | ArcBlock 框架下的可独立部署应用单元 |
| **Node NFT** | 链上节点凭证，用于身份背书和网络发现 |

## 2. 产品愿景与目标

### 2.1 愿景

创建一个**开放、可互操作、多主体参与**的去中心化住宿预订网络，让：
- 每家旅店拥有**真正属于自己的**数字店面
- 多个聚合平台（Hub）**公平竞争**提供服务
- 旅客**自主选择**信任哪个 Hub 或直接信任 Property

### 2.2 核心目标

| 目标 | 衡量指标 |
|------|----------|
| Property 自主权 | 可独立运营、数据可导出、品牌可展示 |
| Hub 可替代性 | Property 可同时入驻多个 Hub，费用可谈判 |
| 信任透明度 | DID 身份 + 链上凭证 + 订单绑定评价 |
| 区域合规 | Hub 可针对本地法规定制审核标准 |
| 创新速度 | 单店可自定义主题、快速实验新功能 |

## 3. 用户角色与场景

### 3.1 角色定义

#### Property Owner（旅店方）

**画像**: 宾馆老板、民宿主、连锁分店经理、青旅运营者

**核心需求**:
- 建立自有品牌官网，不被平台淹没
- 管理房型、价格、日历、订单
- 低成本接入多个分销渠道
- 拥有客户数据和关系

**痛点**:
- OTA 佣金高（15-25%）
- 平台规则限制多
- 品牌弱化，客人记住平台而非店名
- 数据被锁定，难以迁移

#### Hub Operator（聚合运营方）

**画像**: 地区旅游协会、OTA 创业者、垂直领域 KOL、企业差旅服务商

**核心需求**:
- 审核把控入驻 Property 质量
- 通过搜索、推荐、运营获得流量
- 收取订阅费或订单分成盈利
- 建立区域或垂类品牌认知

**痛点**:
- 自建平台技术门槛高
- 获取房源成本高
- 与大平台竞争流量困难

#### Traveler（旅客）

**画像**: 休闲旅行者、商务出差人员、背包客

**核心需求**:
- 发现符合需求的住宿
- 比价、看评价、放心预订
- 灵活的退改政策
- 售后有保障

**痛点**:
- 刷单评价难以辨别
- 图文不符、到店无房
- 平台客服响应慢
- 个人信息被滥用

### 3.2 用户场景

#### 场景 1: 民宿主开设数字店面

**背景**: 张阿姨在大理经营一家有 5 间房的民宿

**流程**:
1. 张阿姨部署 Property Studio Blocklet
2. 配置民宿信息：名称、地址、联系方式
3. 录入 5 个房型，上传照片，设置价格日历
4. 加入「云南民宿联盟」Hub，提交资质审核
5. 审核通过后，房源出现在 Hub 列表中
6. 收到首笔订单，确认接单
7. 客人入住后，收到评价

#### 场景 2: 旅客通过 Hub 预订

**背景**: 小李计划去大理旅游

**流程**:
1. 打开 OpenStay App，选择「云南民宿联盟」Hub
2. 搜索「大理」，筛选「近古城」「可住 2 人」「有 WiFi」
3. 浏览列表，点击张阿姨的民宿查看详情
4. 查看房型、价格、评价、民宿故事
5. 选择日期，确认规则，使用 DID 钱包下单
6. 收到预订确认，查看订单详情
7. 入住后，在 App 中给民宿评价

#### 场景 3: 旅客直接访问 Property

**背景**: 小王通过朋友推荐知道张阿姨的民宿

**流程**:
1. 扫描朋友分享的二维码，直达张阿姨的民宿页面
2. 页面显示「⚠️ 此页面未经 Hub 审核，请自行判断风险」
3. 小王查看民宿信息、照片、历史评价
4. 确认信任后，直接下单预订
5. 订单数据存储在民宿的 Blocklet 中

#### 场景 4: Hub 运营商审核入驻

**背景**: 「云南民宿联盟」Hub 运营团队审核新申请

**流程**:
1. 收到张阿姨的入驻申请，附带营业执照、消防许可
2. 审核员查看 Property Node NFT 质押情况
3. 抽查图片真实性，进行视频验证
4. 批准入驻，设置 tier 为「Pro」
5. Property 数据开始同步到 Hub 索引
6. 收取月度订阅费

## 6. 法律与合规

### 6.1 责任界定

- **Hub 审核模式**: Hub 对审核通过的 Property 承担有限背书责任
- **直连模式**: Hub 不对直连 Property 背书，风险由用户自担
- **Property 直连**: Property 对自有内容负全部责任

### 6.2 合规要求

| 地区 | 要求 |
|------|------|
| 中国大陆 | 旅馆业特种行业许可证、网络安全等级保护 |
| 欧盟 | GDPR 数据保护、民宿短租法规 |
| 美国 | 州/市短租法规、ADA 无障碍 |
